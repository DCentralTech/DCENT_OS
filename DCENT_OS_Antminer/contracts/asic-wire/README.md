# ASIC wire contracts

This directory contains language-neutral, evidence-backed wire vectors. The
CRC5 contract in `v1/bm13xx-command-crc5.json` is sourced from independently
sniffed BM1397 traffic and a separate DCENT_OS live BM1368 command/response
test. Expected trailers are recorded evidence, not values regenerated by the
Rust or Python implementation under test.

The command contract covers host-to-ASIC CRC only. It excludes the `55 aa`
preamble and the CRC trailer. ASIC-to-host trailers are a separate protocol:
`v1/bm13xx-response-crc5.json` records the response-specific Braiins RTL state
machine and independently captured BM1397/BM1362 vectors. The live AM3 BM1362
response ending in `0d` is both a negative command-CRC vector and a positive
response-CRC vector. Never validate it with the command primitive by analogy.

The response contract deliberately stops below hardware identity. AM3-BB's
pre-address GetAddress capture repeats one byte-identical, CRC-valid frame 126
times. That proves response count and family-shaped traffic, but it does not
prove 126 unique chips because the replies carry no distinct address. Xilinx
assigned-address reports provide the payload variation needed to validate the
response CRC algorithm; they do not retroactively add unique addresses to the
AM3 capture.

Legacy BM1387 framing remains unvalidated. Stock FIL code can cover 27 bits of
a four-byte field, while VIL commands cover whole bytes. The Python primitive
supports an explicit `bit_length`, but no destructive or state-changing tool
path should be treated as validated until an independent BM1387 FIL/VIL
command capture pins the complete frame and trailer.

The authored Python implementation lives at
`tools/asic-wire/python/dcentos_asic_wire.py`. Run
`python3 scripts/sync_asic_wire_python.py --write` to refresh the two target
overlay copies, or `--check` to detect drift.
